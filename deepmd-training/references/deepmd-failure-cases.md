# DeePMD-kit Failure Cases

Use this reference when DeePMD-kit data loading, training, freeze/export, testing, model-deviation, or downstream model handoff fails. These cases are diagnostic patterns, not proof that a model is reliable after the immediate issue is fixed.

Some patterns below are informed by public upstream DeePMD-kit issues. Treat those issues as examples of error shapes, software-version boundaries, and diagnostic questions. Do not copy their systems, hyperparameters, descriptor choices, batch sizes, or acceptance criteria as defaults for this repository.

These public GitHub issues are useful as searchable examples of real DeePMD-kit failures:

- Descriptor evaluation out-of-memory behavior: [deepmodeling/deepmd-kit#5507](https://github.com/deepmodeling/deepmd-kit/issues/5507).

Keep the stage boundary clear:

- DeePMD-kit learns an approximate potential-energy surface from DFT labels.
- Training, freeze/export, and test commands produce model artifacts and diagnostics.
- Model-deviation outputs support active-learning and exploration decisions, but do not alone certify production sampling.
- A frozen model is an input to LAMMPS/CHMC/CPIHMC, not the final scientific observable.
- Any fix involving descriptor, cutoff, type map, training systems, loss weights, learning rate, batch size, training steps, seeds, ensemble size, final model choice, or acceptance thresholds requires user confirmation.

## DeePMD Data Or Type Map Is Inconsistent

Typical symptoms:

- Training fails while reading `deepmd/raw`, `deepmd/npy`, or `deepmd/npy/mixed` data.
- Tensor shape mismatch appears during data loading or batching.
- `type_map`, atom names, atom counts, cells, energies, forces, virials, spin, `fparam`, or `aparam` arrays do not match across systems.
- The user wants to train from converted data without inspecting it.

Likely causes:

- Converted data lost labels or optional arrays.
- Systems with incompatible atom counts, type maps, or optional fields were mixed.
- Training input expects a property such as spin, virial, `fparam`, or `aparam` that the data do not contain.
- The DeePMD data were generated by a dpdata route whose source/converted shape was not compared.

Observed upstream example:

- A tensor shape mismatch during training was suspected to involve magnetic spin parameters and data lacking matching spin content: [deepmodeling/deepmd-kit#3949](https://github.com/deepmodeling/deepmd-kit/issues/3949).

Agent response:

1. Route format inspection to `dpdata-format-conversion`.
2. Check frame count, atom count, type map, cells/PBC, energies, forces, virials, and optional arrays before training.
3. Confirm whether `System` or `LabeledSystem` data are appropriate for the training target.
4. Ask the user whether optional properties such as spin, `fparam`, `aparam`, or virials are required.
5. Do not fill missing labels or optional arrays with zeros unless the user provides a documented, valid policy.

## Training Input Hyperparameters Are Missing Or Copied From A Reference

Typical symptoms:

- `input.json` contains `TODO_USER_APPROVAL`.
- Descriptor, cutoff, network architecture, loss weights, learning rate, batch size, stop batch, or seed came from a reference example.
- The user asks the agent to pick DeePMD hyperparameters automatically.
- Training starts but the provenance of data systems, validation split, and type map is not documented.

Likely causes:

- A template or CORR reference example was treated as production input.
- The user has not approved target-system training choices.
- A DP-GEN ensemble training config was confused with optional final specialized training.

Agent response:

1. Stop before producing or running a production `input.json`.
2. List every unapproved hyperparameter and dataset-choice TODO.
3. Treat `templates/reference-examples/corr-deepmd/` as file-organization and log-shape guidance only.
4. Ask whether the model is part of DP-GEN ensemble training or optional final training.
5. Do not choose descriptor, cutoff, architecture, loss weights, learning-rate schedule, batch size, training steps, seeds, or validation split.

## Training Does Not Converge Or Loss Curves Look Abnormal

Typical symptoms:

- `lcurve.out` contains NaN or Inf values.
- Energy, force, or virial RMSE oscillates, grows, plateaus at an unacceptable value, or differs strongly between train and validation columns.
- Repeated runs with the same nominal input show inconsistent convergence behavior.
- `parse_lcurve.py` reports nonmonotonic steps, duplicate steps, missing loss-like columns, inconsistent column counts, or skipped nonnumeric lines.

Likely causes:

- Data contain inconsistent labels, outliers, wrong units, or unconverged DFT labels.
- Descriptor, cutoff, learning rate, loss weights, or batch size are unsuitable for the dataset.
- Training/validation split is not representative.
- Hardware, backend, precision, or software-version behavior differs across runs.

Observed upstream example:

- A reported `se_atten_v2` training case showed random non-convergence and force RMSE oscillation despite similar inputs: [deepmodeling/deepmd-kit#3103](https://github.com/deepmodeling/deepmd-kit/issues/3103).

Agent response:

1. Run or suggest `scripts/parse_lcurve.py` for deterministic first-pass log diagnostics.
2. Inspect data provenance, label quality, units, train/validation split, and type map before changing hyperparameters.
3. Ask the user to approve any changes to descriptor, cutoff, loss weights, learning rate, batch size, stop batch, or seed.
4. Preserve abnormal logs as evidence; do not summarize them as "probably fine".
5. Do not declare a model ready for CHMC/CPIHMC from a completed training command alone.

Useful command:

```bash
python deepmd-training/scripts/parse_lcurve.py PATH_TO_LCURVE_OUT
```

## Data Loading Is Too Slow Or Buffer Warnings Appear

Typical symptoms:

- Logs contain `Data loading buffer is empty or nearly empty`.
- Training progresses but wall time per batch is unexpectedly high.
- GPU utilization is low while data loading or filesystem access is busy.
- The user asks the agent to change worker counts or batch size automatically.

Likely causes:

- Dataset is too fragmented or stored on slow shared storage.
- Loader worker count or prefetch behavior is not appropriate for the environment.
- Batch size, model type, or hardware layout creates a data-loading bottleneck.

Observed upstream example:

- A DPA-2 training report showed repeated data-loading buffer warnings and very slow batches: [deepmodeling/deepmd-kit#3951](https://github.com/deepmodeling/deepmd-kit/issues/3951).

Agent response:

1. Treat performance warnings separately from physical model quality.
2. Ask for DeePMD-kit version, backend, command, dataset location, hardware, worker settings, and batch timing.
3. Consult official help/docs before suggesting version-specific loader options.
4. Do not change batch size or worker counts without user approval.
5. After performance tuning, still require validation/test and reaction-coordinate coverage evidence.

## Backend, GPU, CUDA, Or Custom-Op Errors Occur

Typical symptoms:

- Errors mention CUDA invalid device ordinal, OOM, missing GPU, TensorFlow/PyTorch custom operations, missing gradients, or compiled operator failures.
- `dp train`, `dp freeze`, `dp compress`, `dp test`, or descriptor evaluation works on one machine but fails on another.
- A DP-GEN training task fails repeatedly on the remote worker but not locally.

Likely causes:

- Environment variables or scheduler GPU allocation do not match the requested device.
- DeePMD-kit, TensorFlow/PyTorch, CUDA, compiler, or LAMMPS plugin versions are incompatible.
- A local code modification touched non-differentiable custom ops or unsupported paths.
- Descriptor evaluation needs more memory than ordinary inference.

Observed upstream examples:

- `dp compress` reported `CUDA_ERROR_INVALID_DEVICE` despite train/freeze seeing GPU devices: [deepmodeling/deepmd-kit#3764](https://github.com/deepmodeling/deepmd-kit/issues/3764).
- A modified trainer path triggered `No gradient defined for operation 'ProdEnvMatA'`: [deepmodeling/deepmd-kit#3724](https://github.com/deepmodeling/deepmd-kit/issues/3724).
- A DP-GEN training job repeatedly failed and the remote train log contained TensorFlow errors: [deepmodeling/deepmd-kit#3215](https://github.com/deepmodeling/deepmd-kit/issues/3215).

Agent response:

1. Ask for exact command, backend, DeePMD-kit version, Python version, TensorFlow/PyTorch version, CUDA version, GPU allocation, and scheduler logs.
2. Use official docs or local `-h`/`--help` for version-specific command options.
3. Separate environment/runtime failures from model-science failures.
4. Do not rewrite training or compression commands from memory.
5. If this is inside DP-GEN, route scheduler and task-dispatch diagnosis to `dpgen-active-learning`.

## Descriptor Evaluation Or DeepMD Mixed Data Gives Shape/OOM Problems

Typical symptoms:

- `dp eval-desc` or a descriptor-evaluation command fails on `deepmd/npy/mixed`.
- Descriptor frame count does not match the source system frame count.
- Descriptor extraction OOMs even when ordinary `dp test` inference completes.
- Output descriptors are missing frames, duplicated frames, or cannot be aligned back to source systems.

Likely causes:

- Mixed systems have variable atom counts or frame layout not handled by the command path.
- Descriptor evaluation has different memory behavior from normal testing or inference.
- Batch/chunk behavior is insufficient for the selected descriptor and hardware.

Observed upstream examples:

- `dp eval-desc` reported `ValueError` on `deepmd/npy/mixed` data with multiple frames: [deepmodeling/deepmd-kit#5176](https://github.com/deepmodeling/deepmd-kit/issues/5176).
- Descriptor outputs had a mismatched number of frames on a certain GPU/system path: [deepmodeling/deepmd-kit#5180](https://github.com/deepmodeling/deepmd-kit/issues/5180).
- DPA4-Plus descriptor extraction OOMed even though `dp test` completed: [deepmodeling/deepmd-kit#5507](https://github.com/deepmodeling/deepmd-kit/issues/5507).

Agent response:

1. Do not assume descriptor-evaluation success or failure reflects training-set physical quality.
2. Compare descriptor frame count against source frame count before using descriptors downstream.
3. Ask whether descriptor extraction is actually required for this workflow.
4. If required, check official command options, batching controls, and known issue behavior for the installed version.
5. Do not treat `dp test` success as proof that all auxiliary descriptor tools will run.

## Freeze Or Export Succeeds But Model Readiness Is Overclaimed

Typical symptoms:

- A frozen model file exists, and the user asks whether it is ready for CHMC/CPIHMC production.
- Freeze/export logs show a model was written, but no test or model-deviation evidence is available.
- Multiple frozen models exist and the user asks the agent to pick the best one automatically.

Likely causes:

- Freeze/export portability was confused with scientific reliability.
- Final model selection criteria were not defined.
- Test data or reaction-coordinate coverage evidence is missing.

Agent response:

1. State that freeze/export makes a model portable but does not certify reliability.
2. Ask for freeze checkpoint, frozen model path, training input, training logs, DeePMD-kit version, and test dataset.
3. Use `dp test` or version-appropriate official test command only after the user provides the test dataset and options.
4. Compare candidate models only on the same documented test data and metrics.
5. Do not select a production model without user-approved acceptance criteria.

## `dp test` Results Are Missing, Incomparable, Or Misread

Typical symptoms:

- No `dp test` or equivalent test output is available.
- Models are compared using different test datasets, units, frame counts, or metrics.
- Advanced outputs such as hessian, spin, atomic energy, or atomic virial are involved and the command behavior is version-specific.
- Force error is ignored even though the model will be used for dynamics or CHMC/CPIHMC sampling.

Likely causes:

- Test evidence was skipped after freeze.
- Test datasets were not preserved with provenance.
- The user focused on a single scalar score without reaction-relevant validation.
- Version-specific output interpretation is unclear.

Observed upstream example:

- A code-scan issue flagged caution around reading advanced hessian outputs from `dp test`: [deepmodeling/deepmd-kit#5672](https://github.com/deepmodeling/deepmd-kit/issues/5672).

Agent response:

1. Ask for the test command, test data path, type map, metrics, and software version.
2. Compare energy, force, and virial errors when available.
3. Keep models incomparable if test datasets or units differ.
4. Treat force errors and reaction-coordinate-region errors as critical for downstream sampling.
5. Do not use test results alone to skip model-deviation or reaction-coordinate coverage review.

## Model-Deviation Evidence Is Missing Or Abnormal

Typical symptoms:

- DP-GEN/LAMMPS exploration has no model-deviation output.
- Model-deviation values are high in reaction-coordinate windows or production-like regions.
- The user asks to ignore model deviation because training/test errors look acceptable.
- Model-deviation APIs or tools fail with runtime errors.

Likely causes:

- An ensemble was not trained or not used consistently.
- Trust thresholds were not user-approved.
- Exploration visited regions outside the training distribution.
- Runtime/tooling failure prevented model-deviation calculation.

Observed upstream example:

- A code-scan issue described uncaught C++ exceptions in model-deviation C API helpers: [deepmodeling/deepmd-kit#5622](https://github.com/deepmodeling/deepmd-kit/issues/5622).

Agent response:

1. Route trust-threshold and candidate-selection policy to `dpgen-active-learning`.
2. Ask for ensemble provenance, model paths, exploration conditions, model-deviation files, and trust thresholds.
3. Treat abnormal or missing model deviation as a blocker for production-like CHMC/CPIHMC handoff.
4. Do not infer model-deviation safety from one frozen model or one `dp test` result.
5. Preserve missing evidence as TODO rather than declaring coverage.

## LAMMPS Or Downstream Sampling With A Frozen Model Is Unstable

Typical symptoms:

- LAMMPS trajectories diverge, produce NaNs, or behave non-reproducibly with the DP potential.
- CHMC/CPIHMC sampling enters geometries outside the model's training coverage.
- A frozen model works for short tests but fails under reaction-coordinate exploration.
- The user asks whether a model that passes `dp test` is automatically safe for CPIHMC.

Likely causes:

- Test set does not cover the reaction-coordinate regions.
- Model-deviation evidence is missing for the downstream exploration domain.
- Numerical precision, neighbor-list, hardware, or LAMMPS plugin differences affect trajectories.
- The MLFF is being used outside its valid domain.

Observed upstream example:

- A reproducibility discussion for LAMMPS with a DP potential noted that trajectories can diverge from precision/truncation behavior even with matched seeds: [deepmodeling/deepmd-kit#3270](https://github.com/deepmodeling/deepmd-kit/issues/3270).

Agent response:

1. Route LAMMPS exploration setup to `lammps-exploration`.
2. Ask for model-deviation evidence in the intended reaction-coordinate region.
3. Require user acceptance before CHMC/CPIHMC production sampling.
4. Do not treat exact trajectory reproducibility as the only validity criterion.
5. Do not claim the MLFF is an exact Born-Oppenheimer potential.

## How These Cases Connect To Scripts And Checks

Use `parse_lcurve.py` for conservative log inspection:

```bash
python deepmd-training/scripts/parse_lcurve.py PATH_TO_LCURVE_OUT
```

Use the shared static checker for minimal file-shape checks:

```bash
python common/scripts/check_workflow_files.py \
  --software deepmd \
  --path PATH_TO_TRAINING
```

Use `dpdata-format-conversion` tools before training if data were converted:

```bash
python dpdata-format-conversion/scripts/inspect_dpdata_system.py \
  --input PATH_TO_DATA \
  --format USER_CONFIRMED_FORMAT \
  --labeled
```

All script outputs are diagnostics. DeePMD model readiness still requires user-approved data provenance, training input, validation/test errors, model-deviation evidence, reaction-coordinate coverage, software-version provenance, and downstream acceptance criteria.
